# 루프 엔지니어링

> **TL;DR** — 루프 엔지니어링(Loop Engineering)은 "에이전트에게 프롬프트를 쓰는 그 사람 자리에서 당신을 빼내고, 대신 그 일을 해줄 시스템을 설계하는 것"이다[1]. 여기서 **루프**란 목적만 정해두면 AI가 끝날 때까지 스스로 반복하는 작업 묶음이며[2], **다섯 개 부품(오토메이션·워크트리·스킬·플러그인/커넥터·서브에이전트)과 하나의 기억장치(상태 파일)**로 만든다[1]. Claude Code와 Codex 양쪽 모두 이 부품들을 기본으로 갖췄고, 이름만 다를 뿐 기능은 같다[1][2]. 단, 루프는 일의 형태를 바꿀 뿐 사람을 일에서 빼주지 않는다 — 검증은 여전히 사람 몫이다[1].

## 개요

루프 엔지니어링은 '프롬프트 엔지니어링을 넘어선 다음 단계'로 불리며[3], 지금까지 AI 코딩 에이전트를 쓰는 방식은 기본적으로 **대화**였다. 한 줄 시키고(프롬프트), 답을 읽고, 다시 한 줄 시키는 구조로, 매 턴마다 사람이 운전대를 잡는다[2]. 루프 엔지니어링은 이 운전대를 **시스템에게 넘긴다**. "에이전트가 할 일을 찾아내고, 나눠주고, 검사하고, 끝난 일을 기록하고, 다음 일을 결정하는 작은 시스템을 만들어, 당신 대신 그 시스템이 에이전트를 찌르게(poke) 두는 것"이다[1].

이 용어와 개념을 널리 정립한 사람은 구글에서 14년간 개발자 경험을 이끈 엔지니어링 리더 **Addy Osmani**로, 2026년 6월 7일 자 블로그에서 "Loop engineering is replacing yourself as the person who prompts the agent"라고 정의했다[1]. 이 흐름은 혼자만의 주장이 아니다.

> "이제 코딩 에이전트에 프롬프트를 쓰지 마라. 에이전트에게 프롬프트를 던지는 **루프**를 설계하라." — Peter Steinberger [1]

> "나는 더 이상 Claude에 프롬프트를 쓰지 않는다. Claude에게 무엇을 할지 시키는 루프를 돌리고 있고, 내 일은 그 루프를 짜는 것이다." — Boris Cherny, Anthropic Claude Code 총괄 [1]

루프 엔지니어링은 [[하네스-엔지니어링]]의 "한 층 위"에 자리한다[1]. 하네스가 *하나의 에이전트가 도는 환경*을 만드는 것이라면, 루프는 그 하네스에 **타이머를 달고, 작은 조수들을 띄우고, 스스로 먹이를 주는(feed itself)** 단계까지 올린 것이다[1]. 1년 전만 해도 루프를 짜려면 셸 스크립트를 직접 짜고 계속 관리해야 했지만, 지금은 Claude Code와 Codex 양쪽 모두 부품이 제품 안에 내장됐다[1].

## 본론

### 루프를 이루는 다섯 부품과 기억장치 하나

루프는 다섯 가지 핵심 부품과, 거기에 더해지는 하나의 기억장치로 만들어진다[1]. 양쪽 제품이 같은 부품을 갖추고 있어 이름만 다를 뿐이다.

| 부품 | 역할 | Codex | Claude Code |
| --- | --- | --- | --- |
| **① 오토메이션** | 정해진 주기로 발견·분류(트리아지) | Automations 탭(프로젝트·프롬프트·주기·환경 지정), Triage inbox, `/goal` | cron·hooks·GitHub Actions, `/loop`(주기 반복), `/goal`(조건 만족까지) |
| **② 워크트리** | 병렬 작업 충돌 격리 | 스레드별 내장 워크트리 | `git worktree`, `--worktree` 플래그, 서브에이전트 `isolation: worktree` |
| **③ 스킬** | 프로젝트 지식 코드화 | `SKILL.md`(`$이름` 또는 암묵 호출) | `SKILL.md`(동일 형식) |
| **④ 플러그인·커넥터** | 외부 도구 연결 | Connectors(MCP) + 플러그인(배포용) | MCP 서버 + 플러그인 |
| **⑤ 서브에이전트** | 만드는 AI / 검사하는 AI 분리 | `.codex/agents/`(TOML), 요청 시 병렬 후 합치기 | `.claude/agents/`, 에이전트 팀 |
| **기억장치 상태** | 끝난 일·남은 일 기억 | 마크다운 또는 Linear(커넥터) | 마크다운(`AGENTS.md`·진행 파일) 또는 Linear(MCP) |

- **① 오토메이션**은 루프를 "한 번 돌린 것"이 아니라 "실제 루프"로 만드는 심장이다[1]. 발견한 실행은 사람의 수신함(triage inbox)으로 올라오고, 아무것도 못 찾은 실행은 스스로 아카이브된다[2]. OpenAI는 사내에서 이걸 매일 이슈 분류, CI 실패 요약, 커밋 브리핑, "지난주 누군가 심어둔 버그 사냥" 같은 지루하지만 끝없이 반복되는 일에 돌린다[1][2]. 두 제품 모두 `/goal`이라는 공통 원시(primitive)를 제공하는데, 이것은 "모든 테스트가 통과하고 린트가 깨끗해질 때까지" 같은 검증 가능한 조건이 참이 될 때까지 매 턴 뒤 *별도의 작은 모델*이 완료 여부를 검사하는 식으로 돈다[1].
- **② 워크트리**는 에이전트 둘이 같은 파일을 동시에 덮어쓰는 실패를 막는다. 같은 저장소 이력을 공유하면서 각자 별도 브랜치의 독립 폴더를 쓰게 해, 한쪽의 수정이 다른 쪽 체크아웃에 닿지 않는다[1]. 다만 충돌은 막아줘도 "동시에 몇 개를 돌릴 수 있느냐"를 정하는 건 결국 **사람의 검토 대역폭**이다[1][2].
- **③ 스킬**은 매 세션마다 프로젝트를 처음부터 다시 설명하는 수고를 없앤다[1]. 에이전트는 세션이 바뀌면 맥락이 백지가 되고, 빈 구멍을 자신 있게 추측으로 메워버리므로, 지식을 `SKILL.md`에 적어두면 매 작업마다 다시 읽고 시작한다[2]. *스킬이 '작성하는 형식'이라면, 플러그인은 그걸 '배포하는 방식'*이다[1][2].
- **④ 플러그인·커넥터**는 에이전트의 손발이다. 커넥터(보통 [[MCP]]로 구현)가 에이전트를 이슈 트래커·DB·API·Slack에 연결하고, 플러그인은 그 커넥터와 스킬을 한 묶음으로 포장해 동료가 한 번에 설치하게 해준다[2]. 이 덕분에 에이전트가 "고치세요"라고 말로 끝내는 게 아니라 직접 PR을 열고 티켓을 연결하고 CI가 통과하면 채널에 알리는 식으로 *실제 환경 안에서 행동*한다[1].
- **⑤ 서브에이전트**는 루프 구조에서 가장 유용한 장치로, **코드를 짠 모델이 자기 결과를 후하게 평가하는 편향**을 깨기 위해 '만드는 AI'와 '검사하는 AI'를 일부러 나누는 것이다[1]. 역할마다 모델과 추론 강도를 달리 둘 수도 있다 — 보안 검토용은 강한 모델에 깊은 추론으로, 빠른 탐색용은 가벼운 읽기 전용 모델로 굴리는 식이다[2].
- **기억장치(상태 파일)**는 대화 컨텍스트 *바깥에* 오래 남는 파일로, 뭘 시도했고·뭐가 통과했고·뭐가 남았는지를 적어둔다. Addy Osmani의 한마디가 핵심이다 — **"The agent forgets, the repo doesn't(에이전트는 잊어도, 저장소는 잊지 않는다)."**[1]

### 실제로 돌아가는 루프 한 바퀴

부품들을 모으면 한 바퀴는 **발견 → 분류(트리아지) → 수정 → 검증**의 흐름으로 압축된다[2].

1. 오토메이션이 매일 아침 저장소에서 한 번 돈다[1].
2. 깨어난 루프가 트리아지 스킬을 불러 어제의 CI 실패·열린 이슈·최근 커밋을 읽고 결과를 마크다운(또는 Linear 보드)에 적는다[1][2].
3. 처리할 만한 작업마다 격리된 워크트리를 열고, 한 서브에이전트가 수정안을 쓰면 *두 번째 서브에이전트*가 프로젝트 스킬과 기존 테스트를 기준으로 그 수정안을 검토한다[1][2].
4. 커넥터가 PR을 열고 티켓을 갱신한다[2].
5. 루프가 못 다룬 것은 사람 수신함으로 올라오고, 아무것도 못 찾은 실행은 스스로 아카이브된다[1][2].
6. 상태 파일이 이 전체의 중심으로, 다음 날 작업이 멈춘 지점부터 이어지게 한다[1].

Addy Osmani의 표현을 그대로 옮기면 — *"You designed it one time. You did not prompt any of those steps."(당신은 이걸 한 번 설계했을 뿐, 그 단계들 중 어느 것도 직접 시키지 않았다.)*[1][2]

### 루프는 사람을 일에서 빼주지 않는다

분위기를 한 톤 낮춰야 한다. 루프는 *일의 형태를 바꿀 뿐, 사람을 일에서 빼주지 않는다*. 오히려 루프가 좋아질수록 사람이 붙잡아야 할 부분이 더 선명해진다[1][2]. Addy Osmani는 세 가지를 못 박는다.

1. **검증은 여전히 사람 몫이다.** 사람이 안 볼 때 돌아가는 루프는 곧 사람이 안 볼 때 실수하는 루프다. "당신의 일은 *작동을 직접 확인한* 코드를 내보내는 것이다." 검토 없이 루프에 다 맡기면 제품 품질이 무너져 악순환에 빠진다고, 저자 본인도 인정한다[1][2].
2. **이해도는 노력하지 않으면 부식한다.** 루프가 안 쓴 코드를 빨리 만들어낼수록, *읽지 않으면* 내가 들고 있는 코드가 뭔지 모르게 된다. 루프의 산출물을 읽는 일은 선택이 아니다[1].
3. **가장 큰 위험은 '인지적 항복'이다.** 똑같은 루프를 두 사람이 만들어도 정반대 결과가 나온다 — 한 명은 *깊이 아는 일을 더 빨리 하려고* 쓰고, 다른 한 명은 *그 일을 이해하지 않으려고* 쓴다[1][2]. **"The loop doesn't know the difference. You do.(루프는 그 차이를 모른다. 당신은 안다.)"**[1]

갓대희의 자기 실험은 이 경고의 증거가 된다. 같은 글을 Claude가 스스로 채점하면 9.04점, Codex가 독립으로 채점하면 7.43점이 나와 **1.6점**이 벌어졌다 — 자기가 만든 결과를 자기가 후하게 본다는 뜻이다. 이 수치는 한 번의 하네스 런에서 관찰한 값이며 공개 벤치마크가 아니라는 한계를 단다[2].

그러면서도 저자는 균형추를 단다. 루프를 짜되, 에이전트에게 직접 프롬프트를 던지는 방식도 여전히 효과적이며, 결국 "올바른 균형을 찾는 일(It's all about finding the right balance)"이라고 마무리한다[1].
