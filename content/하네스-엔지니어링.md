# 하네스 엔지니어링

> **TL;DR** — 하네스 엔지니어링(Harness Engineering)은 AI 코딩 에이전트를 둘러싼 시스템 — 에이전트가 보는 것·할 수 있는 것·검사받는 방식·실패 후 살아남는 법·반복되는 실수가 구조적 수정으로 바뀌는 길 — 을 설계하는 실천이다[1]. 핵심 주장은 **"모델이 병목이 아니라 하네스가 병목이다"**[1]이며, 같은 모델로 하네스만 바꿔서 벤치마크 순위가 도약한 사례(LangChain 52.8%→66.5%[1], 단일 모델 6.7%→68.3%[1])가 이를 뒷받침한다. 다섯 가지 우려(Context·Constraints·Verification·Recovery·Feedback loop)로 정리되며[1], [[루프-엔지니어링]]은 이 하네스의 "한 층 위"에 해당한다[2].

## 개요

승마에 비유하면 이해가 쉽다. 말(AI 모델)이 아무리 강력해도 고삐와 안장(하네스)이 없으면 원하는 방향으로 달릴 수 없다[4]. 코딩 에이전트도 같아서, Claude Code든 Cursor든 Codex든 **모델 자체보다 그 모델을 감싸는 시스템이 결과를 결정**한다[4].

이 용어는 **Mitchell Hashimoto**(HashiCorp 창립자이자 Terraform 창시자)가 2026년 2월 자 개인 블로그에서 명명했다[5][3]. 그는 AI 에이전트를 느리고 회의적으로 도입하며, 에이전트가 실수할 때마다 *그 실수가 다시는 발생하지 않도록* 영구적인 수정을 시스템에 새겨넣는 습관을 거의 무의식적으로 들인 과정을 적었다[2]. 한 줄로 줄이면 이렇다.

> "하네스 엔지니어링은 에이전트가 같은 실수를 두 번 다시 하지 않도록 **주변 설정을 고치는 것**이다. 에이전트가 틀렸을 때 채팅에서 고치고 넘어가는 대신, 그 실수가 앞으로 불가능해지도록 설정을 바꾼다. 수정은 모델 바깥에 살기 때문에 남는다." — Sumit Pandey의 정리 [3]

다시 말해, 더 좋은 모델을 기다리지 말고 **모델을 둘러싼 시스템을 고치라는 것**이다[4]. Addy Osmani는 이것을 [[루프-엔지니어링]]과 구분해, 하네스가 *하나의 에이전트가 도는 환경*을 만드는 단계이고 루프가 그 위에 타이머와 자동 먹이 공급을 얹는 단계라고 설명한다[2].

### 왜 모델보다 하네스가 중요한가

숫자가 모든 걸 말해준다[4].

- **LangChain**: 같은 모델(GPT-5.2-Codex)로 **하네스만 바꿔서** Terminal Bench 2.0에서 52.8% → 66.5%, 30위권 밖에서 Top 5로 도약했다[1][6]. 13.7점 점프. 모델은 처음부터 끝까지 그대로였다[1].
- **OpenAI Codex 팀**: 2026년 2월 발표한 5개월 실험에서 엔지니어 3명이 에이전트를 몰아 약 **100만 줄, 1,500개 PR**을 사람이 쓴 코드 한 줄 없이 배포했다[1]. 포스트모템은 하네스 문헌의 독립적 확인처럼 읽힌다 — "초기 진척이 예상보다 느렸던 건 Codex가 무능해서가 아니라 **환경이 덜 정의돼 있어서(underspecified)**였다"[1].
- **Can Bölük**: 코딩 벤치마크에서 *편집 도구 형식만* 바꾸고 모델은 그대로 둬서 한 설정을 6.7%에서 68.3%로 올렸다 — 약 10배 개선, 모델 변경 없음[1].
- **Vercel**: 내부 d0 데이터 에이전트에서 **도구의 80%를 제거**했더니 작업 성공률이 80%에서 100%로 올랐다[1]. 능력을 빼니 결과가 좋아진 것.
- **Stripe**: Minions 시스템으로 **주 1,000개 이상의 PR**을 배포하며, 에이전트당 400개 넘는 MCP 도구 전체가 아니라 큐레이션된 부분 집합만 노출하는 식으로 과도한 도구가 성능을 깎는 문제를 해결한다[1].
- **ETH Zurich 연구**: 300개 SWE-bench 태스크 + 138개 AGENTbench 태스크로 검증해, 잘 설계된 하네스가 모델 성능을 좌우한다[4].

세 팀, 세 아키텍처가 같은 결론에 도달한다 — **모델이 병목이 아니라 하네스가 병목이다**[1]. 모델은 이제 상품(commodity)이라, 모든 팀이 Claude Code와 Codex를 가진다면 차별점은 모델이 아니라 *비계(scaffolding)*에 있다[1].

## 본론

### 에이전트 개발의 세 계층

하네스 엔지니어링은 대부분의 팀이 순서대로 밟는 길의 세 번째 정거장이다[1].

1. **프롬프트 엔지니어링 (2022–2024)**: 레버는 모델 앞에 놓인 텍스트 — 표현, few-shot 예시, 사고사슬(chain-of-thought) 구조. 한 번 교환에 한 번 출력. 질문은 *"더 좋은 답을 얻으려 뭘 말해야 하나"*[1].
2. **컨텍스트 엔지니어링 (2024–2025)**: 세션이 길어지고 도구 호출이 시끄러워지며 질문이 바뀐다 — 문서 검색, 이력 압축, 어떤 도구 결과를 어떻게 포맷할지, 창이 찰 때 뭘 버릴지. 질문은 *"모델이 지금 당장 뭘 볼 수 있어야 하나"*[1].
3. **하네스 엔지니어링 (현재)**: 모델이 긴 작업을 감당할 만큼 유능해졌지만 여전히 프로덕션에선 불안정해지자, 질문이 또 바뀐다. 이제 입력 자체가 아니라 전체 **운전 범위(operational envelope)** — 도구, 가드레일, 검증, 복구, 영속 메모리 — 이 대상이 된다. 질문은 *"모델이 실제 일을 하려면 주변에 뭐가 필요한가"*[1].

각 층은 바로 아래 층을 **흡수(subsume)**한다. 프롬프트 엔지니어링은 컨텍스트 엔지니어링 안에, 컨텍스트 엔지니어링은 하네스 엔지니어링 안에 산다[1]. 운영체제 비유를 빌리면 — 모델은 CPU, 컨텍스트 창은 RAM, **하네스는 OS**, 에이전트는 그 위에서 도는 애플리케이션이다. 모델이 `.env`를 읽어도 되는지 따지는 게 아니라 OS가 시스템콜을 그냥 거부하는 식이다[1].

### 다섯 가지 우려 (The Five Concerns)

하네스 문헌을 읽다 보면 다섯 주제가 반복해 등장한다. 각각 감사 렌즈로 쓸 수 있다[1].

| 우려 | 정의 | 대표적 실패 양상 |
| --- | --- | --- |
| **① Context (맥락)** | 에이전트가 행동 전 읽는 것 | 조작(fabrication), 산문 비대 |
| **② Constraints (제약)** | 절대 하지 말아야 할 것 | 파괴적·되돌릴 수 없는 행동 |
| **③ Verification (검증)** | 행동 뒤 작업을 검사하는 법 | 조용한 회귀, 미검증 주장 |
| **④ Recovery (복구)** | 실패 후 상태가 살아남는 법 | 압축·크래시 뒤 맥락 상실 |
| **⑤ Feedback loop (피드백)** | 반복 실수가 영구 수정으로 가는 길 | 같은 버그, 다른 날 |

- **① Context**: 에이전트가 과제에 들고 가는 지도다. 너무 얇으면 조작하고, 너무 두꺼우면 잠긴다. 1,000줄짜리 지시 파일은 '맥락으로 위장한 함정'이다 — 847번째 줄에 묻힌 핵심 제약을 놓치고, 파일은 아무도 유지 안 하는 속도로 썩는다[1]. 좋은 맥락은 *가늘고 현재*여야 한다. **단일 거대 매뉴얼보다 깊은 진실원을 가리키는 짧은 입구**가 항상 이긴다[1].
- **② Constraints**: 행동 *전*에 에이전트를 이끄는 결정론적 규칙 — 린터, deny-훅, 필수 지시 섹션, 권한 경계. 모델이 따져야 할 필요가 없게 해, **추론보다 싸다**[1]. 핵심 프레이밍은 *구현이 아니라 불변량(invariant)을 강제하라* — "경계에서 데이터를 파싱하라"는 요구하되 어떤 라이브러리로 파싱할지는 에이전트가 고르게 두는 식[1]. 덜 눈에 띄는 종류의 제약은 *안 주는 것*이다. 도구를 너무 많이 주면 '도구 선택 혼란(tool thrash)'이 생겨 무엇을 쓸지 결정하느라 비용을 낭비하므로, **2~3개로 시작**하고 실제 한계에 부딪힐 때만 추가하는 게 좋다[1][4].
- **③ Verification**: 에이전트가 자기 작업을 증명하도록 구조적으로 검사하는 단계 — 타입체크, 편집 뒤 파일 다시 읽기, 빌드가 통과하지 않으면 성공을 반환하지 않는 post-action 훅[1]. 모델은 스스로 작업을 개선할 수 있지만 **스스로 루프를 시작하진 않는다**. LangChain의 트레이스 분석이 발견한 가장 흔한 실패는 에이전트가 해법을 쓰고 자기 코드를 다시 읽고 "괜찮아 보인다"며 멈추는 것 — 테스트를 한 번도 안 썼다는 점이다[1]. 모델이 검증 *능력*은 있어도 루프를 강제하는 건 하네스의 몫이다[1]. 또 같은 에이전트가 만든 것의 최종 판사로는 약하므로, GAN처럼 생성자와 평가자를 갈라 적대적 긴장으로 품질을 끌어올리는 것이 권장된다[1].
- **④ Recovery**: 세션 내구성이다. 컨텍스트 창이 차거나 에이전트가 작업 중 크래시할 때 어떻게 되는가. 복구가 없는 하네스는 뭔가 흥미로운 일이 생기자마자 진척을 잃는다[1]. Cognition 팀이 문서화한 **'맥락 불안(context anxiety)'**은 묘한 실패 양상이다 — 창이 차는 것을 의식한 모델이 지름길을 택해 과제를 일찍 끝내버리는 것. 그들의 수정은 순수 하네스 엔지니어링이었다: 1M-토큰 베타를 켜되 *실제 사용은 200K로 제한*하고, 모델에게 여유가 많다고 속인다. 불안이 사라졌다. 모델 변경 없이 환경만 더 똑똑하게[1].
- **⑤ Feedback loop**: 모든 실수를 다음 세션의 입력으로 바꾸는 영속 메모리 — footguns, lessons, 결정, 세션 로그[1]. 이것 없으면 매주 화요일 에이전트는 같은 함수를 새 파일에 다시 쓰고, 당신은 매번 잡아낸다[1]. 두 따라름이 있다: (1) **맥락에 없으면 존재하지 않는다** — Slack 스레드나 선임의 머릿속에만 있는 아키텍처 합의는 새 세션에 보이지 않으므로 저장소에 영속되어야 한다[1]. (2) **영속된 메모리는 썩는다** — 교훈이 낡으므로, 성숙한 하네스는 더 이상 현실을 반영하지 않는 항목을 통합·제거하는 정기 '가꾸기(garden pass)'를 포함한다[1].

### 실천 레버 — 무엇부터 손댈까

한국어 정리는 하네스를 다섯 가지 실천 레버로 풀어, 막상 적용하려는 사람에게 길잡이를 준다[4].

1. **시스템 프롬프트(`CLAUDE.md` / `AGENTS.md`)**: 저장소 료트에 두는 마크다운으로 매 세션 시작 시 컨텍스트에 주입된다. **60줄 이하**, 범용 지침만, 디렉터리 구조 나열 금지, 조건부 규칙 금지, AI 생성 내용 금지(ETH Zurich 연구: AI 생성 파일은 성능을 오히려 깎고 추론 비용을 약 20% 올린다)가 핵심 규칙이다[4].
2. **스킬(점진적 지식 공개)**: 모든 지식을 시스템 프롬프트에 넣는 대신 태스크별로 모듈을 나눠, 에이전트가 필요할 때만 불러오게 한다. 이를 **점진적 공개(Progressive Disclosure)**라 부르며, 컨텍스트 창이 깨끗해져 혼란을 막는다[4].
3. **MCP 서버**: [[MCP]](Model Context Protocol)로 외부 도구를 연결한다. 단 너무 많으면 tool thrash가 생기므로 2~3개로 시작할 것[4].
4. **서브에이전트(컨텍스트 방화벽)**: "프론트엔드 에이전트/백엔드 에이전트" 식의 역할 분담이 아니라, **독립된 검증자**를 두는 데 의미가 있다[4].
5. **피드백 루프**: 반복되는 실수를 저장소에 영속해 같은 버그가 재발하지 않게 하는 것[4].

### 왜 복리로 작용하는가, 그리고 한계

하네스는 두 방향으로 복리로 쌓인다[1]. 첫째 **세션 간** — 좋은 모델 + 좋은 하네스는 좋은 세션을 주고, *다음 세션마다 조금씩 더 나아진다*(피드백 루프가 잡은 것, 메모리가 저장한 것 때문). 둘째 **모델 릴리스 간** — 손으로 짠 제어 흐름을 에이전트에 과하게 박아넣은 팀은 업그레이드마다 시스템이 부서지지만, 얇은 하네스는 다음 모델이 나와도 CPU만 갈아끼운다[1].

중요한 뉘앙스: **원칙은 옮겨가도 튜닝은 옮겨가지 않는다.** 구조적 변경(더 나은 검증 루프, 더 가는 맥락 관리)은 모델을 바꿔도 깔끔하게 전이되지만, 정확한 프롬프트 표현이나 정확한 도구 설명은 그렇지 않다. LangChain이 Codex로 66.5%를 받은 *같은 설정*을 Claude Opus에 돌리니 59.6%가 나왔다. 모델마다 하네스를 다시 다듬어야 한다[1]. 이런 이유로 잘 만든 하네스의 모든 구성 요소는 '더 똑똑한 모델이 와서 없앨 것'을 기다리고 있다 — **삭제될 것을 전제로 만들라(build components that are designed to be deleted)**[1].
